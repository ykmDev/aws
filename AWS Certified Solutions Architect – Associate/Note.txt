                        explanation:
                            One-Time vs Persistent Request

                            This is the most important part.

                            There are two request types.

                            ① One-Time Request

                            Imagine you tell AWS:

                            I need one Spot Instance.

                            AWS finds one.

                            Spot Request
                                │
                                ▼
                            Launch EC2

                            Later, the instance is interrupted or terminated.

                            What happens?

                            Nothing.

                            The request is finished.

                            AWS does NOT launch another EC2.

                            Example:

                            Request
                                │
                            Launch EC2
                                │
                            Interrupted
                                │
                            Finished
                            Myanmar

                            One-Time ဆိုတာ

                            တစ်ခါပဲ တောင်းတာ

                            Instance ပျက်သွားရင်

                            AWS က အသစ်ပြန်မဖန်တီးဘူး။
                            ② Persistent Request

                            Now imagine you tell AWS:

                            I always want one Spot Instance running.

                            AWS launches one.

                            Spot Request
                                │
                                ▼
                            EC2 Running

                            If AWS interrupts it:

                            EC2 Interrupted
                                │
                                ▼
                            AWS automatically tries
                            to launch another one

                            As soon as capacity becomes available, AWS starts a replacement.

                            Myanmar

                            Persistent ဆိုတာ

                            အမြဲတမ်း EC2 ရှိနေချင်တယ်။

                            တစ်လုံးပျက်သွားရင်

                            AWS က အသစ်တစ်လုံး
                            ထပ်ရှာပြီး run ပေးမယ်။

                            Step 3: Why Can't I Just Terminate the EC2?

                            Suppose you have:

                            Persistent Request
                                    │
                                    ▼
                            Spot EC2

                            You terminate the EC2 manually.

                            What happens?

                            AWS sees:

                            The request still exists!

                            So AWS launches another EC2.

                            Terminate EC2
                                │
                                ▼
                            AWS creates new EC2

                            Many beginners are surprised by this behavior.

                            To Stop Everything

                            You must:

                            Cancel the Spot Request
                            Then terminate the Spot Instance (if it's still running)

                            Otherwise AWS continues trying to satisfy the request.

                            Step 4: What Does This Mean?

                            Your note says:

                            You can only cancel spot instance requests that are open, active, disabled.

                            These are the request states.

                            Open

                            AWS is still looking for capacity.

                            Need EC2
                            Searching...

                            No instance yet.

                            Active

                            AWS found capacity.

                            Spot Request
                                │
                                ▼
                            EC2 Running
                            Disabled

                            The request exists, but AWS isn't launching instances right now.

                            Think of it as paused.

                            Only these request states can be cancelled.

                            If a request is already completed or cancelled, there's nothing left to cancel.

                            Real Example

                            Suppose you create:

                            Persistent Request

                            1 x t3.medium

                            AWS launches:

                            EC2 #1

                            AWS later interrupts it.

                            EC2 #1
                            Stopped

                            AWS automatically launches:

                            EC2 #2

                            You don't want any more Spot Instances.

                            Wrong way:

                            Terminate EC2 #2

                            AWS immediately launches:

                            EC2 #3

                            Correct way:

                            Cancel Persistent Request

                            Then
                            Terminate EC2
                            Now AWS stops creating new instances.

        Private vs Public IP (IPv4) Fundamental Differences
            Public IP:
                Public IP means the machine can be identified on the internet (WWW) 
                    Public IP ဆိုတာ Internet ပေါ်ကနေ တိုက်ရိုက်ရှာတွေ့နိုင်တဲ့ IP 
                Must be unique across the whole web (not two machines can have the same public IP).
                Can be geo-located easily
            Private IP:
                Private IP means the machine can only be identified on a private network only
                    Private Network အတွင်းမှာပဲ အသုံးပြုတဲ့ IP
                The IP must be unique across the private network
                BUT two different private networks (two companies) can have the same IPs.
                    Because they are different private networks. They never see each other.
                Machines connect to WWW using a NAT + internet gateway (a proxy)
                    using flow:
                        Private EC2
                            │
                            ▼
                        NAT Gateway
                            │
                            ▼
                        Internet Gateway
                            │
                            ▼
                        Internet
                        Imagine:
                            Private EC2
                            10.0.1.10

                            It wants to install updates.

                            Destination: ubuntu.com

                            But

                            10.0.1.10
                            is private.

                            Internet doesn't know where it is.
                        NAT means: Network Address Translation
                            It changes:
                            Private IP
                            ↓
                            Public IP
                        Example
                            Before NAT
                            10.0.1.10

                            After NAT
                            54.12.90.8

                            Internet sees:
                            54.12.90.8

                            not
                            10.0.1.10
                        Without NAT
                            Private EC2
                            10.0.1.10
                            ↓
                            Internet
                            ❌ Doesn't know this IP
                        Important:
                            A Private EC2 can:
                            ✅ Access Internet (through NAT Gateway)

                            But
                            ❌ Internet cannot start a connection to the Private EC2.

                            This is why private subnets are much more secure.

        First, Why Do We Need Placement Groups?
            Normally, when you launch EC2 instances:

            EC2-1
            EC2-2
            EC2-3

            AWS decides where to place them.

            You don't control:

            Which physical server
            Which rack
            How close they are

            AWS automatically chooses.

            Sometimes, however, you have special requirements.

            For example:

            Requirement 1
            I want all EC2s
            to communicate
            VERY FAST.

            OR

            Requirement 2
            I don't want
            all EC2s to fail
            at the same time.

            AWS asks:

            "How should I place your EC2 instances?"

            That's exactly what Placement Groups are for.
        သတ်မှတ်ပေးတဲ့ Strategy: 3
            Cluster Placement Group
                Keyword: Performance
                aws pusts ec2: Very close together. communication becomes extremely fast.
                EC2 တွေကို
                တစ်နေရာတည်း
                အနီးနားမှာထားတာ။
                Disadvantage:
                    Imagine Rack A catches fire.
                    Everything fails together.
                    Higher performance,
                    Lower fault tolerance (အမှားခံနိုင်ရည်နည်းခြင်း)
                Note: Very safe.
            Spread Placement Group
                Keyword: Availability
                aws pusts ec2: separates EC2 instances.
                EC2 တွေကို
                တစ်နေရာတည်းမှာ မထားဘူး။
                ဝေးဝေးခွဲထားတယ်။
                Note: very safe.
            Partition Placement Group
                Each partition uses completely different racks.
                Partition Placement Group ဆိုတာ
                EC2 တွေကို
                အုပ်စုကြီးတွေခွဲထားတာ။
                Many Instances
                ↓
                One Partition
                Applications like:
                Hadoop, HDFS, HBase,
                Apache Kafka
                Cassandra,
                Hundreds of EC2
                already work in partitions.
                AWS aligns the hardware with the application design.
                Note: Scalable + fault isolation(ချို့ယွင်းချက်ခွဲထုတ်ခြင်း။)
            Memory:
                | Strategy      | Goal                               | Best For                    | Risk                                       | 
                | ------------- | ---------------------------------- | --------------------------- | ------------------------------------------ |
                | **Cluster**   | Lowest latency                     | HPC, ML, Trading            | One rack failure can affect many instances |
                | **Spread**    | Highest availability               | Critical applications       | Max 7 instances per AZ                     |
                | **Partition** | Fault isolation for large clusters | Hadoop, Cassandra, Kafka,Big Data| One partition can fail, others continue    |
                HPC means high-performance computing.

    Hibernate:
        3 important (lifecycle)actions:
            1. Stop
                After Stop
                    RAM
                    -------------
                    ❌ Cleared

                    EBS
                    -------------
                    ✅ Still Exists
                When Start again
                    AWS does:
                    Boot Operating System
                    ↓
                    Start Services
                    ↓
                    Start Application

                    Everything starts from the beginning.
                If EC2 is stopped,
                RAM is cleared.
                Redis cache disappears (unless configured otherwise).
                When EC2 starts,
                it must rebuild the cache. caches get warmed up, and that can take time!
                This causes slower startup.
                    Cache Warm-up:
                        our application uses Redis.
                            Initially

                            Cache

                            Empty

                            First user visits.

                            Database
                            ↓
                            Redis

                            Now cache becomes populated.
                            This is called Cache Warm-up.
            2. Hibernate
                Think of Windows.
                Instead of Shut Down,
                you click:
                Hibernate
                Computer powers off.
                BUT
                before shutting down,
                Windows saves RAM to disk.
                AWS does exactly the same.
                Before Hibernate
                    RAM
                        Application
                        Browser
                        Variables
                        Sessions
                    AWS copies RAM -> EBS
                    Then powers off.
                    When Start again
                        AWS does
                        Read RAM
                        ↓
                        Restore Memory
                        ↓
                        Continue
                        Application continues exactly where it stopped.
                Why is Hibernate faster?
                    Because OS is not stopped / restarted, aws perform: OS -> frozen(Pause) instead of shutdown.
                    Normal Start
                        Boot OS
                        ↓
                        Start Application
                        ↓
                        Load Cache
                        ↓
                        Reconnect DB
                        ↓
                        Ready
                        Maybe 60 seconds
                    Hibernate
                        Power On
                        ↓
                        Restore RAM
                        ↓
                        Continue
                        Maybe 15 seconds
                Is our application still running?
                    No.
                    Power is OFF.
                    CPU is OFF.
                    RAM has no power.
                    Nothing is executing.
                Why must Root EBS be encrypted?
                    Root EBS is the boot disk.
                        Imagine RAM contains
                            Password
                            API Keys
                            JWT Token
                            Database Password
                            User Session
                        AWS saves RAM
                        ↓
                        EBS

                        If EBS wasn't encrypted
                        someone with disk access could potentially read all those secrets.
            3. Terminate
                Terminate means
                    Destroy. RAM and Root EBS Deleted. Everything is gone.
        Compare All Three:
            Stop
                RAM ❌ Lost
                EBS ✅ Keep
            Hibernate
                RAM ✅ Saved (preserved=မပျောက်အောင် သိမ်းထားတာ)
                EBS ✅ Keep
            Terminate
                RAM ❌ Lost
                EBS ❌ Lost

            many beginners think: "All EC2 storage = EBS"
                ❌ Not true.

                1. EBS (Network Storage)
                2. Instance Store (Physical Storage)
            • EBS volumes are network drives with good but "limited" performance
            • If you need a high-performance hardware disk, use EC2 Instance Store.
                Because: The disk is physically attached to the host machine.
                example:

                        Your PC
                            ↓
                            Network
                            ↓
                            Storage
                        Because it is over the network:
                        Some latency exists.

                        Your PC
                        ↓

                or

                or

                Need maximum speed.

                    EBS
                    Stop EC2
                    ↓
                    Data Survives

                    Instance Store
                    Stop EC2
                    ↓
                    Data Lost

                For EBS:
                    Snapshot
                    AWS helps.

                For Instance Store:
                    You handle backup
                    Your application must replicate data elsewhere.

                        HTTPS
                        │
                        ▼
                        EC2

                        In this case:

                        ✅ The browser talks directly to the EC2 instance.
                        ✅ EC2 must have an SSL/TLS certificate installed (for example, in Nginx or Apache).
                        ✅ EC2 decrypts the HTTPS traffic itself.

                        Example:

                        Client
                            │
                        HTTPS (443)
                            │
                            ▼
                        Nginx (SSL Certificate)
                            │
                        HTTP
                            │
                        Laravel

                        So EC2 receives HTTPS on port 443, and Nginx terminates SSL.
                    Case 2: With an Application Load Balancer (Most Common)

                        Architecture:

                        Browser
                            │
                        HTTPS
                            │
                            ▼
                        ALB
                            │
                        HTTP
                            │
                            ▼
                        EC2

                        Here:

                        The SSL certificate is attached to the ALB (often using AWS Certificate Manager).
                        The ALB decrypts the HTTPS request.
                        The EC2 instance receives plain HTTP.

                        This is called: SSL Termination

                        This is the default architecture used by many AWS applications.

                            ▼
                        ALB
                            │
                        HTTPS
                            ▼
                        EC2

        Sticky Sessions:
            supported on three LB: CLB, ALB, NLB.
            User က Login လုပ်တယ်။ Login Data ကို EC2-A မှာ သိမ်းထားတယ်။ 
            နောက် Request တစ်ခုမှာ ALB က EC2-B ဆီ ပို့လိုက်ရင် EC2-B မှာ Login Data မရှိဘူး။ User ကLogout ဖြစ်သွားနိုင်တယ်။
            Instead of randomly choosing every request, Every request goes to the same server.
            That's called Session Affinity or Sticky Session.
            To remember, AWS sends a cookie to the browser.
                Example:
                Set-Cookie: AWSALB=xxxxxxxx
                Browser stores it.
                Next request:
                    Cookie: AWSALB=xxxxxxxx
                ALB reads it and says: "Oh! This user belongs to EC2-A."
                Then forwards the request to EC2-A.
                The cookie doesn't live forever.
            
            Is Sticky Session the Best Solution?
            AWS says:
                Not really.
                Instead,
                store sessions centrally. like: redis.
                Other options include storing sessions in a shared database.
            အားနည်းချက်
                Enabling stickiness may bring imbalance.
                Load Balance မညီနိုင်ဘူး
                Modern cloud applications usually avoid relying on sticky sessions.
                Instead:
                    Users
                    ↓
                    ALB
                    ↓
                    EC2-A
                    ↓
                    shared session store (Redis)

                    EC2-B
                    ↓
                    shared session store (Redis)

                Any server can handle any request.
                This scales much better.
            Cookie Names:
                Cookie Name Must Be Unique
                two cookie have:
                    Duration-based Cookies:
                        LB creates. not do developer. AWSALB for alb, AWSELB for clb.
                            Duration means 1 hr, 2 hr, 1 day. after that time, expires -> alb can choose another ec2.
                    Application-based Cookie:
                        your application creates. as like as laravel_session, PHPSESSID.
                        Never use AWSALB, AWSALBAPP, or AWSALBG as your application's cookie names.
                Which One Should We Use?
                    Most companies use
                    Duration-based
                    because it's simpler.
                    Application-based is used only when the application already has its own session cookie and you want ALB to use it.
