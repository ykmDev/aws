                        explanation:
                            One-Time vs Persistent Request

                            This is the most important part.

                            There are two request types.

                            ① One-Time Request

                            Imagine you tell AWS:

                            I need one Spot Instance.

                            AWS finds one.

                            Spot Request
                                │
                                ▼
                            Launch EC2

                            Later, the instance is interrupted or terminated.

                            What happens?

                            Nothing.

                            The request is finished.

                            AWS does NOT launch another EC2.

                            Example:

                            Request
                                │
                            Launch EC2
                                │
                            Interrupted
                                │
                            Finished
                            Myanmar

                            One-Time ဆိုတာ

                            တစ်ခါပဲ တောင်းတာ

                            Instance ပျက်သွားရင်

                            AWS က အသစ်ပြန်မဖန်တီးဘူး။
                            ② Persistent Request

                            Now imagine you tell AWS:

                            I always want one Spot Instance running.

                            AWS launches one.

                            Spot Request
                                │
                                ▼
                            EC2 Running

                            If AWS interrupts it:

                            EC2 Interrupted
                                │
                                ▼
                            AWS automatically tries
                            to launch another one

                            As soon as capacity becomes available, AWS starts a replacement.

                            Myanmar

                            Persistent ဆိုတာ

                            အမြဲတမ်း EC2 ရှိနေချင်တယ်။

                            တစ်လုံးပျက်သွားရင်

                            AWS က အသစ်တစ်လုံး
                            ထပ်ရှာပြီး run ပေးမယ်။

                            Step 3: Why Can't I Just Terminate the EC2?

                            Suppose you have:

                            Persistent Request
                                    │
                                    ▼
                            Spot EC2

                            You terminate the EC2 manually.

                            What happens?

                            AWS sees:

                            The request still exists!

                            So AWS launches another EC2.

                            Terminate EC2
                                │
                                ▼
                            AWS creates new EC2

                            Many beginners are surprised by this behavior.

                            To Stop Everything

                            You must:

                            Cancel the Spot Request
                            Then terminate the Spot Instance (if it's still running)

                            Otherwise AWS continues trying to satisfy the request.

                            Step 4: What Does This Mean?

                            Your note says:

                            You can only cancel spot instance requests that are open, active, disabled.

                            These are the request states.

                            Open

                            AWS is still looking for capacity.

                            Need EC2
                            Searching...

                            No instance yet.

                            Active

                            AWS found capacity.

                            Spot Request
                                │
                                ▼
                            EC2 Running
                            Disabled

                            The request exists, but AWS isn't launching instances right now.

                            Think of it as paused.

                            Only these request states can be cancelled.

                            If a request is already completed or cancelled, there's nothing left to cancel.

                            Real Example

                            Suppose you create:

                            Persistent Request

                            1 x t3.medium

                            AWS launches:

                            EC2 #1

                            AWS later interrupts it.

                            EC2 #1
                            Stopped

                            AWS automatically launches:

                            EC2 #2

                            You don't want any more Spot Instances.

                            Wrong way:

                            Terminate EC2 #2

                            AWS immediately launches:

                            EC2 #3

                            Correct way:

                            Cancel Persistent Request

                            Then
                            Terminate EC2
                            Now AWS stops creating new instances.

        Private vs Public IP (IPv4) Fundamental Differences
            Public IP:
                Public IP means the machine can be identified on the internet (WWW) 
                    Public IP ဆိုတာ Internet ပေါ်ကနေ တိုက်ရိုက်ရှာတွေ့နိုင်တဲ့ IP 
                Must be unique across the whole web (not two machines can have the same public IP).
                Can be geo-located easily
            Private IP:
                Private IP means the machine can only be identified on a private network only
                    Private Network အတွင်းမှာပဲ အသုံးပြုတဲ့ IP
                The IP must be unique across the private network
                BUT two different private networks (two companies) can have the same IPs.
                    Because they are different private networks. They never see each other.
                Machines connect to WWW using a NAT + internet gateway (a proxy)
                    using flow:
                        Private EC2
                            │
                            ▼
                        NAT Gateway
                            │
                            ▼
                        Internet Gateway
                            │
                            ▼
                        Internet
                        Imagine:
                            Private EC2
                            10.0.1.10

                            It wants to install updates.

                            Destination: ubuntu.com

                            But

                            10.0.1.10
                            is private.

                            Internet doesn't know where it is.
                        NAT means: Network Address Translation
                            It changes:
                            Private IP
                            ↓
                            Public IP
                        Example
                            Before NAT
                            10.0.1.10

                            After NAT
                            54.12.90.8

                            Internet sees:
                            54.12.90.8

                            not
                            10.0.1.10
                        Without NAT
                            Private EC2
                            10.0.1.10
                            ↓
                            Internet
                            ❌ Doesn't know this IP
                        Important:
                            A Private EC2 can:
                            ✅ Access Internet (through NAT Gateway)

                            But
                            ❌ Internet cannot start a connection to the Private EC2.

                            This is why private subnets are much more secure.

            Elastic IPs:
                when stop/start ec2 instance -> change public ip.
                if need fixed IP, will use Elastic IP. it never changes until you release it.
                Elastic IP is pubilc IPv4 IP.
                can attach it to one instance.
                With an Elastic IP address, you can mask the failure of an instance or software by rapidly remapping the address to another instance in your account.
                    "mask the failure" means enduser don't even know the server changed.
                each aws account can have 5 Elastic IP by default. you can ask AWS to increase that.
                Overall, try to avoid using Elastic IP:
                    They often reflect poor architectural decisions.
                        Why is Elastic IP considered a poor architecture?
                            Imagine a company has:
                            100 EC2 Servers

                            Do you need:
                            100 Elastic IPs?

                            No.

                            That's difficult to manage.

                            AWS recommends:
                            One Load Balancer
                            ↓
                            Many EC2
                            instead.
                            Better Solution is
                                use a random public IP and register a DNS name to it.
                                    Example
                                        EC2:
                                        Public IP
                                        18.181.10.20

                                        Instead of telling users:
                                        18.181.10.20

                                        You create DNS:
                                        www.mycompany.com
                                        ↓
                                        18.181.10.20

                                        If Public IP changes,
                                        just update the DNS record.

                                        Users still type:
                                        www.mycompany.com

                                        They never notice.
                            Or, use a Load Balancer and don't use a public IP.
                            Example
                                Users
                                │
                                ▼
                                Load Balancer
                                │
                                ┌─┴───────────┐
                                ▼             ▼
                                EC2-1       EC2-2
                                Users never connect directly to EC2.
                                They connect only to the Load Balancer.
                                If EC2 fails,
                                the Load Balancer sends traffic to another EC2 automatically.
                                No need for Elastic IPs on each EC2.
                                Imagine Facebook.
                                    Do they give every server an Elastic IP?

                                    No.

                                    Millions of users connect to:
                                    facebook.com
                                    ↓
                                    DNS
                                    ↓
                                    Load Balancer
                                    ↓
                                    Thousands of servers
                Elastic IP = One stable address for one instance (or quickly moved to another instance).
                Load Balancer = One stable endpoint for many instances.

    Placement Groups:
        Placement Group ဆိုတာ
        EC2 Instance တွေကို
        AWS Data Center ထဲမှာ
        ဘယ်လိုနေရာချမလဲ
        ဆိုတာ သတ်မှတ်ပေးတဲ့ Strategy ဖြစ်တယ်။
        First, Why Do We Need Placement Groups?
            Normally, when you launch EC2 instances:

            EC2-1
            EC2-2
            EC2-3

            AWS decides where to place them.

            You don't control:

            Which physical server
            Which rack
            How close they are

            AWS automatically chooses.

            Sometimes, however, you have special requirements.

            For example:

            Requirement 1
            I want all EC2s
            to communicate
            VERY FAST.

            OR

            Requirement 2
            I don't want
            all EC2s to fail
            at the same time.

            AWS asks:

            "How should I place your EC2 instances?"

            That's exactly what Placement Groups are for.
        သတ်မှတ်ပေးတဲ့ Strategy: 3
            Cluster Placement Group
                Keyword: Performance
                aws pusts ec2: Very close together. communication becomes extremely fast.
                EC2 တွေကို
                တစ်နေရာတည်း
                အနီးနားမှာထားတာ။
                Disadvantage:
                    Imagine Rack A catches fire.
                    Everything fails together.
                    Higher performance,
                    Lower fault tolerance (အမှားခံနိုင်ရည်နည်းခြင်း)
                Note: Very safe.
            Spread Placement Group
                Keyword: Availability
                aws pusts ec2: separates EC2 instances.
                EC2 တွေကို
                တစ်နေရာတည်းမှာ မထားဘူး။
                ဝေးဝေးခွဲထားတယ်။
                Note: very safe.
            Partition Placement Group
                Each partition uses completely different racks.
                Partition Placement Group ဆိုတာ
                EC2 တွေကို
                အုပ်စုကြီးတွေခွဲထားတာ။
                Many Instances
                ↓
                One Partition
                Applications like:
                Hadoop
                Kafka
                Cassandra,
                Hundreds of EC2
                already work in partitions.
                AWS aligns the hardware with the application design.
                Note: Scalable + fault isolation(ချို့ယွင်းချက်ခွဲထုတ်ခြင်း။)
            Memory:
                | Strategy      | Goal                               | Best For                    | Risk                                       | 
                | ------------- | ---------------------------------- | --------------------------- | ------------------------------------------ |
                | **Cluster**   | Lowest latency                     | HPC, ML, Trading            | One rack failure can affect many instances |
                | **Spread**    | Highest availability               | Critical applications       | Max 7 instances per AZ                     |
                | **Partition** | Fault isolation for large clusters | Hadoop, Cassandra, Kafka,Big Data| One partition can fail, others continue    |
