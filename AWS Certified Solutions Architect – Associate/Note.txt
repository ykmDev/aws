Access aws:
    1) management console(protected by password + MFA)
    2) cli (protected by access keys)
    3) sdk (for code, protected by access keys)
    Access key ID ~= username
    Secret Access Key ~= password
IAM:
    IAM Security Tools:
        iam credential report (account-level):
            a report of lists all acc users and status of their credentials
        iam Last Accessed in iam user detail (old name: access advisor)(user-level):
            service permissions granted to a user. and service who last accessed.
    IAM Guidelins Best Practices:
        one physical user = one aws user
        assign user to group, and assign permissions to group.
        use and enforce MFA.
        create and use Roles for giving permissions to AWS services.
        Audit permissions of your account using credential report & Last accessed.
        ** Never share iam users & access keys **
    Summary:
        Users: physical users, has a password for console.
        Groups: contains users only.
        Policies: json doc that outlines permissions for users to groups.
        Roles: EC2 instances or aws services.
        Security: MFA + password Policy.
        cli: manage services using command line.
        sdk: manage services using programming language.

EC2:
    Security Groups:
        as a "firewall" on ec2 instances.
        control how traffic is allowed into or out of ec2.
        only contain allow rules,
        can reference by IP or by Security group.
        Security Groups are VPC scoped.(Security Groups are locked down to a Region/VPC combination)
            Security Group တစ်ခုဟာ
            Region တစ်ခု + VPC တစ်ခု အတွင်းမှာပဲ အသုံးပြုနိုင်တယ်။
                eg:
                    သင်မှာ:
                        Region: Tokyo (ap-northeast-1)
                        VPC: Production-VPC
                        ရှိတယ်ဆိုပါစို့။
                        Security Group:
                        WebServer-SG
                        ကို create လုပ်လိုက်တယ်။
                        ဒီ Security Group ကို:
                        ✅ Tokyo Region ထဲက
                        ✅ Production-VPC ထဲက
                        EC2 တွေမှာပဲ attach လုပ်နိုင်တယ်။
        Does live "outside" the EC2 - if traffic is blocked the EC2 instance won't see it.
            Security Group က EC2 instance အတွင်းမှာ မဟုတ်ဘဲ
            EC2 အပြင်ဘက်မှာ firewall အနေနဲ့ အလုပ်လုပ်တယ်။

            Traffic တစ်ခုကို Security Group က block လုပ်လိုက်ရင်
            EC2 instance ထိတောင် မရောက်ဘူး။
                e.g:
                EC2 တစ်လုံးရှိတယ်:

                EC2 Web Server
                IP: 10.0.1.10

                Security Group Rule:

                Allow:
                HTTP (80) from anywhere

                NO SSH (22)

                    ↓

                    ↓
                3. EC2

                SSH (22) ?

                instance level firewall

                လို့ ခေါ်ပေမယ့်

                EC2 Operating System ထဲမှာ run နေတာ မဟုတ်ဘူး

        How can administrators securely access EC2 instances in a private subnet?
        ans: Bastion Host.
            Bastion Host =  SSH ဝင်ဖို့အတွက် သီးသန့်ထားတဲ့ EC2 server
            Without Bastion
                Internet
                ↓
                Web1 (SSH open)
                App1 (SSH open)
                DB1 (SSH open)
                Server အားလုံး Port 22 ဖွင့်ထားရမယ်။
                ❌ Security မကောင်း
            With Bastion Host
                Internet
                ↓
                Bastion Host (Public Subnet)
                ↓ SSH
                Private EC2 Servers
            Admin က:
            Bastion Host ကို SSH ဝင်မယ်
            Bastion Host ကနေ Private EC2 တွေကို SSH ထပ်ဝင်မယ်
